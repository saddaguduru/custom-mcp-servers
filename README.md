# custom-mcp-servers

Custom-built MCP Servers
A suite of MCP (Model Context Protocol) servers built for Quality Engineering and SDLC workflows, enabling AI assistants to capture network traffic, generate test plans, and produce API specifications — all through natural language.
MCP Servers:
All servers run on a remote machine and communicate with the AI assistant via HTTP transport. Prompts are defined as /.github/prompts/*.prompt.md files (not @mcp.prompt() decorators).
Server	Port	Public Tools	Description
DevTools to HAR	8001	10	Captures live network traffic from Chrome/Edge via CDP; returns HAR 1.2 JSON content
JMeter	8002	7	Creates JMeter 5.4.3 / OpenJDK 11 test plans (.jmx) from HAR content or OpenAPI 3.0.3 specs; returns JMX strings
OpenAPI	8003	6	Generates OpenAPI 3.0.3 specs from Spring Boot source code or HAR traffic; merges and validates specs
Rest Assured	8004	5	Generates Java API functional test frameworks (Cucumber + Rest Assured + TestNG) from OpenAPI specs or HAR traffic
WireMock	8006	5	Converts HAR traffic into WireMock 3.9.1 stub mapping files (.json) with response body files
Postman	8007	9	Generates Postman Collection v2.1 JSON, environments, and Newman CI/CD configs from HAR or OpenAPI
Playwright	8008	6	Generates Playwright + TypeScript test projects with page objects from HAR or OpenAPI
Contract Testing	8009	6	Generates Pact v4 contracts and Spring Cloud Contract Groovy DSL from OpenAPI or HAR

Note: Each server includes a health-check tool (included in the counts above) that returns server name, version, status, and uptime.


Quick Start:
Automated Setup (recommended)
# Clone the repository
git clone <repository-url>
cd qe-mcp-servers
# One-time setup (venv + dependencies + firewall rules)
.\scripts\setup.ps1
# Start all servers (background jobs, bind to 0.0.0.0 for remote access)
.\scripts\start-servers.ps1
# Stop all servers
.\scripts\stop-servers.ps1
Manual Setup
# Create virtual environment
python -m venv .venv
.venv\Scripts\Activate.ps1
pip install -r requirements.txt
# Start individual servers (default: localhost only)
$env:MCP_BIND_HOST = "0.0.0.0"   # Set to bind on all interfaces for remote access
python devtools_to_har_mcp.py   # DevTools        → http://0.0.0.0:8001/mcp
python jmeter_mcp.py            # JMeter          → http://0.0.0.0:8002/mcp
python openapi_mcp.py           # OpenAPI         → http://0.0.0.0:8003/mcp
python restassured_mcp.py       # Rest Assured    → http://0.0.0.0:8004/mcp
python wiremock_mcp.py          # WireMock        → http://0.0.0.0:8006/mcp
python postman_mcp.py           # Postman         → http://0.0.0.0:8007/mcp
python playwright_mcp.py        # Playwright      → http://0.0.0.0:8008/mcp
python contract_test_mcp.py     # Contract Testing→ http://0.0.0.0:8009/mcp

Architecture (Remote-Server Model):


Content-based I/O: Tools accept content strings as input and return content/results as strings. HAR-processing tools also accept a har_file_path parameter for same-machine scenarios, avoiding large inline payloads. For very large HAR files in remote deployments, use client/send_har.py to call tools directly from the terminal. The AI assistant handles all local file reads/writes.
Cross-Server Workflows:
The servers are designed to work together. JMeter, OpenAPI, Rest Assured, WireMock, Postman, Playwright, and Contract Testing are downstream consumers of .har file artifacts produced by the DevTools to HAR server — they depend on .har files but have no dependency on the DevTools to HAR server at runtime. Several servers can additionally consume OpenAPI specs produced by the OpenAPI server:


Note: JMeter, OpenAPI, Rest Assured, WireMock, Postman, Playwright, and Contract Testing can all consume .har files exported manually from the Chrome DevTools Network tab. The OpenAPI server can additionally generate specs directly from Java Spring Boot source code (no .har required). JMeter, Rest Assured, Postman, Playwright, and Contract Testing servers can also consume OpenAPI specs generated by the OpenAPI MCP Server.
Exclusion Configuration:
All traffic-processing servers (DevTools to HAR, JMeter, OpenAPI, WireMock, Postman, Playwright, Contract Testing) share a common set of exclusion rules via lib/config_loader.py. In the remote-server model, exclusion rules are defined in /.github/copilot-instructions.md and passed to tools via the exclude_config_json parameter. When omitted, built-in defaults are used.
Filter Category	Purpose
Non-HTTP schemes	Requests using blob:, data:, chrome-extension: schemes are always rejected (code-level)
excluded_domains	Hostnames always excluded from capture/conversion
excluded_path_patterns	URL path substrings always excluded (case-insensitive)
excluded_http_methods	HTTP methods always excluded (e.g. OPTIONS)
excluded_urls	Full URLs always excluded (exact match)
excluded_extensions	Static resource file extensions excluded
excluded_content_types	Static resource MIME types excluded

See /.github/copilot-instructions.md for the full exclusion config JSON that the AI assistant passes to tools.
Typical end-to-end workflows”
1.Capture network traffic with the DevTools to HAR server, or export manually from Chrome DevTools (F12 → Network → Export HAR — useful in restricted Virtual Desktop without admin privileges)
2.Create a JMeter test plan from the .har file or from an OpenAPI spec
3.Generate an OpenAPI spec from the .har file or from Spring Boot source code
4.Merge OpenAPI specs from multiple sources (source code + HAR) for a complete API contract
5.Validate the generated OpenAPI spec for structural correctness
6.Generate WireMock stub mappings from the .har file for mock server testing
7.Create a Postman collection and Newman CI/CD config from the .har file or OpenAPI spec
8.Generate Playwright + TypeScript test projects from the .har file or OpenAPI spec
9.Generate Pact v4 contract tests or Spring Cloud Contract DSL from the OpenAPI spec or HAR traffic

Understanding Tools vs Prompts:
	Tools	Prompts
What they do	Execute a single action and return data	Provide a multi-step workflow template for the AI
Where they live	Defined in *_mcp.py with @mcp.tool()	/.github/prompts/*.prompt.md files
How to use	Ask naturally; the AI picks the right tool	Select from the prompt picker (type /) or ask by name
Example	"List all open Chrome tabs"	"Use the capture_har_from_chrome prompt"

Guardrails & Security:
All servers include built-in guardrails for production readiness:
Feature	Description
Rate Limiting	Token-bucket rate limiter (default 60 req/min per server)
Payload Size Validation	Rejects oversized payloads (default 50 MB)
Path Traversal Protection	File paths validated against allowed directories
Structured Logging	JSON-formatted logs with per-request correlation IDs
Standardized Responses	Consistent {"status": "success/error", ...} envelope format
Error Sanitization	Internal file paths stripped from error messages
Health Check	Every server exposes a health_check tool (server name, version, status, uptime)
Graceful Shutdown	SIGINT/SIGTERM handlers with cleanup logging
Localhost Binding	Default bind host is 127.0.0.1 (configurable via MCP_BIND_HOST)

Environment Variables:
Variable	Default	Purpose
MCP_LOG_LEVEL	INFO	Logging verbosity (DEBUG, INFO, WARNING, ERROR)
MCP_LOG_FORMAT	json	Log output format (json or text)
MCP_MAX_PAYLOAD_MB	50	Maximum payload size in MB
MCP_ALLOWED_FILE_DIRS	(cwd)	Comma-separated list of allowed file directories
MCP_RATE_LIMIT	60	Maximum requests per minute per server
MCP_BIND_HOST	127.0.0.1	Network interface to bind to (0.0.0.0 for remote access)

Note: The start-servers.ps1 script automatically sets MCP_BIND_HOST=0.0.0.0 so remote clients can connect. When starting servers manually for remote access, set $env:MCP_BIND_HOST = "0.0.0.0" before launching.
